# 系统核心业务流程规划

> **项目摘要与核心理念**
>
> 本系统以健康守护为核心，面向学生、班级管理员（教职工）及系统管理员三类角色，构建多角色权限体系。
>
> 1. **动态化问卷机制**：支持学生按规范填写体温及症状等信息，系统结合自动化的“因子评分算法”初步判定风险等级，实现信息的标准化采集与结构化存储。
> 2. **双维度数据模型**：在系统设计中引入“空间维度（区域→楼栋→房间）”与“组织维度（班级）”相结合的数据模型。学生既绑定具体宿舍房间，又隶属于固定班级，为后续交叉分析与精细化隔离打下基础。
> 3. **智能预警规则引擎**：设计基于时间窗口与数量阈值的规则引擎，通过定时任务对健康上报数据进行聚合分析。满足预设条件时自动向对应班级管理员推送通知，实现从“被动响应”向“主动监测”的转变。
> 4. **严格的数据权限隔离**：系统管理员可进行全局配置、审计及数据维护；班级管理员仅可基于其所负责班级的范围查看学生健康数据与趋势统计，保障数据访问的安全性。

---

## 1. 🎓 学生侧：信息闭环与自主打卡

### 1.1 注册与登录

- **注册**：学生录入基础信息（学号、真实姓名、手机号等），后端校验学号唯一性，成功后默认分配“学生”角色（写入 `user_role`）。
- **登录与鉴权**：系统发放 Token，记录 `user` 的班级 (`class_id`) 和房间 (`room_id`)。

### 1.2 症状自查与健康上报（核心逻辑）

- **前端填报**：学生通过表单提交信息，为保证数据真实性，填报时无需手动输入个人基础信息，后端直接自动提取 Token 绑定用户身份及其所属班级/房间。包含以下核心数据：
  - **体温** (`temperature`)
  - **症状** (`symptom_json`，多选，配置有“无症状”联动排他选项)
  - **确诊疾病** (`disease_id`，基于最近高发的传染病字典 `infectious_disease` 连表下拉动态获取，默认含“无疾病”选项)
  - **接触史** (`exposure_history`，描述是否接触过疫区或疑似感染人员)
  - **疫苗接种情况** (`vaccination_status`：已完成接种/部分接种/未接种)
- **后端风险评估（因子评分算法）**：
  系统不再依赖管理员主观判断，而是采用**动态因子评分算法**。后端接收数据后，结合提交的各项指标（如发烧体温权重高，存在接触史加分，已接种疫苗减分，选定的疑似传染病严重程度 `severity_level`）计算综合得分。
  最终将总得分映射为风险等级（`risk_level`：1低、2中、3高），并将整条评估结果落盘存入 `health_report` 表。
- **联动预警判定**：
  在数据落盘的同时，异步触发预警系统。比对 `warning_rule` 中定义的针对全校或该班级的规则（如 24 小时内某班级触发 3 次中高风险上报），一旦越限，即向 `warning_record` 与 `warning_record_detail` 写入告警记录，通知班级管理员。

### 1.3 信息查阅与通知接收

- **历史查询**：读取个人历史健康上报数据，前端图表渲染风险趋势。
- **通知下发接收**：学生定期查询自身所属班级（`class_id`）相关的系统通知及全校通知。**已读/未读状态将完全由前端利用本地缓存（LocalStorage / IndexedDB）实现**，减少服务器并发写入压力。

## 2. 👨‍🏫 班级管理员侧：班级管理与预警干预

### 2.1 班级数据总览与监测

- **权限绑定与多日看板**：系统依据 `manager_class` 中的管理关系实现严格的权限隔离。班级管理员可通过下拉菜单，切换并查看其所负责班级在任意指定日期的健康上报数据；而系统管理员则可俯瞰全校所有班级。
- **智能比对查漏**：看板界面通过将班级人员全量名单与当天填报记录自动联表比对，不仅呈现了已打卡学生的具体体温、风险及症状，更能直观地过滤并标记出**“未填报”**的人员名单，极大地便利了辅导员的日常追踪与督促。

### 2.2 预警信息跟进与处理

- **告警接收**：查询涉及其管理班级的 `warning_record` 信息。
- **事件追溯**：管理员可点击具体预警，通过 `warning_record_detail` 下钻查看究竟是哪几位学生的哪几次健康上报触发了该规则。
- **闭环操作**：核实并采取干预措施（如要求隔离或就医）后，将系统中的预警记录状态变更为“已处理”。

### 2.3 精准通知发布

- 管理员仅能向其所管辖的班级发送系统通知，发布时选择指定的班级或者发往全校。前端通过时间倒序展示给班级学生。系统通知的已读/未读状态将完全由前端利用本地缓存（Cookie / LocalStorage）维护，避免后端并发读写压力。

## 3. ⚙️ 系统管理员侧：全局控制与规则调配

### 3.1 传染病字典管理（动态基座）

- **业务作用**：管理员可增删改查 `infectious_disease` 表，维护系统当前的传染病类型库。配置字段包括疾病严重程度、潜伏期及对应症状等。
- **数据支撑**：该字典是健康上报时学生下拉选择的数据源，也是后端进行“因子评分算法”计算风险等级的重要权重依据。

### 3.2 组织架构与账号管理

- **基础架构**：维护“区域->楼栋->房间”的层级结构（`area` 表），以及“学院->专业->班级”的基础信息（`class` 表）。
- **人员与资源分配**：支持在班级与楼栋房间的对应层级中，通过全局带分页与条件过滤的搜索机制，高效、防卡顿地将大批学生/用户绑定分配至指定班级或房间。
- **用户与角色权限 (RBAC)**：管理系统所有账号，灵活配置角色与权限（菜单/按钮级），支持在班级管理中精准筛选出“班级管理员”角色库的人员，以直接配置其下辖班级。

### 3.3 预警规则动态引擎

- **规则定义**：维护 `warning_rule` 表，设定预警触发条件（例如：“流感”在“软工一班”，时间窗口 48 小时，人数阈值 5 人）。
- 可按传染病种类或班级维度进行细化配置，使防御网更具针对性。

### 3.4 审计日志管理

- 监控 `operation_log`，审查重点操作（如修改传染病严重程度、批量重置密码）的操作人与来源 IP，实现系统的全面追踪与安全维稳。
